# 交付记录规范（每个开发 agent 完成任务后必须执行）

完成一张任务卡后，**必须做以下三件事**，否则视为未完成：

## 1. 在任务卡末尾追加「✅ 完成记录」
在对应的 `docs/tasks/T0-X.md` 文件末尾，按下面模板追加一段。

## 2. 在 `docs/DEVLOG.md` 顶部加一行
格式：`- [YYYY-MM-DD] T0-X 完成 — 一句话摘要（agent 名）`

## 3. 提交信息引用任务ID
如：`git commit -m "T0-3: implement FeishuClient (add/update/list/get_due)"`

---

## 完成记录模板（复制填写）

```markdown
## ✅ 完成记录
- **任务**：T0-X 〈标题〉
- **状态**：已完成 / 部分完成（说明）
- **完成日期 / 负责 agent**：YYYY-MM-DD / 〈名字〉
- **实现摘要**：做了什么（3-5 句）
- **改动文件**：
  - 新增：path/to/file
  - 修改：path/to/file（改了什么）
- **接口变更**：无 ／ 有（列出改了哪个签名、为什么、影响哪些任务、是否已通知）
- **新增依赖 / 配置**：requirements 改动？需要哪些 .env 项 / 飞书表？
- **测试**：新增/通过的测试；`pytest -q` 结果
- **自验收报告**（按 docs/ACCEPTANCE.md 该任务清单，逐条，必填）：

  | 验收项 | 验证命令/步骤 | 实际结果(证据) | 通过 |
  |---|---|---|---|
  | … | … | … | ✅/⚠️ |

  > 任一 ⚠️ → 标记"部分完成"并说明，不得标记已完成。
- **数据来源**（若产出含数据结论）：
- **已知限制 / 遗留 TODO**：
- **解锁的下游任务**：完成后哪些任务现在可以开始
```

---

## 为什么要这样记
- **可追溯**（系统红线 NFR-2/NFR-3）：谁、何时、改了什么、为什么，全程留痕。
- **不打架**：接口变更必须显式记录，下一个 agent 才不会被悄悄改的签名坑到。
- **可接力**：下一个 agent 看完成记录就能接手，不用回头猜上下文。
